# SVM Model Implementation

This package contains the active classical-model implementation for attribution,
profiling, profiling-signal extraction, diagnostics, and feature importance.

## Package Map

| Path                                         | Role                                                                                |
| -------------------------------------------- | ----------------------------------------------------------------------------------- |
| [training/](training/)                       | Direct attribution, stacked attribution, profiling classifiers, and staged dispatch |
| [signals/](signals/)                         | Predicted profiling matrices and oracle profiling matrices                          |
| [diagnostics/](diagnostics/)                 | Attribution diagnostics, profiling-transfer checks, and stylometry audits           |
| [importance/](importance/)                   | Direct and stacked feature-importance analysis                                      |
| [linear_svm_common.py](linear_svm_common.py) | Shared matrix loading, feature assembly, fitting, and prediction helpers            |

## Phase Responsibilities

Direct attribution (`training/train_svm_attribution.py`) trains `LinearSVC`
author classifiers from materialized sparse blocks such as `char`, `word`, and
`stylo`. Development chooses candidates on validation folds; final evaluation
uses frozen development selections.

Stacked attribution (`training/train_stacked_attribution.py`) trains one
calibrated `LinearSVC` per feature family and a `LogisticRegression` top model.
The top model's training matrix is generated by inner cross-validation so each
outer-train speech receives base-model probabilities from models that did not
see that speech. Final outer-fold base models then produce validation or test
meta-features with the same column structure. Predicted or oracle profiling
blocks, when configured, are appended at this top-model level.

Profiling (`training/train_profiling_classifiers.py`) trains calibrated
`LinearSVC` classifiers for profile targets such as party, gender, age bin, and
party bloc. These profilers are trained on background authors, not selected
attribution authors. Development training saves one calibrated model per
profiling fold. Final profiling training fits a final bundle on the full
profiling corpus using each target's selected development feature layout.

## Predicted Profiling Signals

`signals/profiling_signal_extractor.py` applies saved profiling classifiers to
raw attribution speeches and writes derived sparse matrices into attribution
materializations. The extractor rebuilds the profiler input space rather than
reusing attribution-materialized `char`, `word`, or `stylo` matrices.

For development units, an attribution fold uses the profiling fold with the same
unit id. The extractor loads vectorizers and scalers from that profiling
materialization fold. For final test units, it loads the final profiling bundle
saved by final profiling training.

Text blocks use the saved profiling `TfidfVectorizer.transform()` methods.
Stylometry uses raw attribution stylometry from `row_features`, aligned by
`id_speech` to each attribution unit's row order, then transformed with the
profiling fold or final stylometry scaler. Attribution-materialized
`X_*_stylo.npz` files are not used because they were scaled in the attribution
feature space.

The extractor writes these blocks for each role:

```text
X_{role}_profiling.npz
X_{role}_profiling_{target}.npz
X_{role}_profiling_hard.npz
X_{role}_profiling_hard_{target}.npz
```

Probability and hard-label matrices come from the same profiler probability
outputs. The hard-label representation is the one-hot argmax version. Both are
registered as derived blocks in the attribution materialization manifest.

## Oracle Profiling Signals

`signals/ground_truth_signal_injector.py` writes upper-bound oracle variants
using true profile labels instead of classifier probabilities. Oracle files use
`profiling_oracle*` block names so they can coexist with predicted profiling
outputs:

```text
X_{role}_profiling_oracle.npz
X_{role}_profiling_oracle_{target}.npz
```

Oracle runs are diagnostics for how much the Phase 3 architecture could gain
with perfect profile information. They are not predicted-profile runs.

Direct module CLIs in this package are for focused maintenance and assume
prerequisite split, feature, materialization, and profiling artifacts already
exist.
